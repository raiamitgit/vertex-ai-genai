# ServiceNow Model Context Protocol (MCP) Server 🚀

This repository provides a sandbox demonstration, fully authenticated Model Context Protocol (MCP) server that integrates **ServiceNow** directly with **Gemini Enterprise** (and other LLMs supporting the StreamableHTTP transport).


It implements a unified host that serves both the **Mock OAuth 2.0 Consent Handshake** and the **JSON-RPC 2.0 MCP endpoints** on a single port, making it highly optimized for containerized deployment on **Google Cloud Run**.

---

## 📦 Features & Supported Actions

The server exposes **6 comprehensive database tools** for managing ServiceNow tickets:

| Action Name | Description | Supported Parameters |
|---|---|---|
| `list_incidents` | Lists recent incidents in chronological order. | `limit` (default: 5) |
| `query_incident` | Retrieves complete ticket details (caller, priority, assigned to, urgency, etc.). | `ticket_id` (e.g., `INC0010019`) |
| `get_ticket_comments` | Retrieves full conversation history and user journal entries. | `ticket_id` |
| `add_ticket_comment` | Posts a new user-facing comment/update to a ticket. | `ticket_id`, `comment_text` |
| `delete_ticket_comment` | Retracts/deletes a specific comment record by its sys_id. | `comment_sys_id` |
| `update_ticket_fields` | Modifies a ticket's description or its lifecycle state. | `ticket_id`, `description`, `state` (1-8) |

## 🛠️ Local Installation & Development

### 1. Clone the repository and install dependencies:
```bash
pip install -r requirements.txt
```

### 2. Configure your environment:
Copy the example credentials file and populate it with your ServiceNow Personal Developer Instance (PDI) credentials:
```bash
cp .env.example .env
```
Open `.env` and edit:
```env
SN_INSTANCE_URL=https://your-instance.service-now.com
SN_USERNAME=admin
SN_PASSWORD=your-password
```

### 3. Start the local development server:
```bash
uvicorn src.main:app --host 0.0.0.0 --port 8080
```
Verify the server is active by opening `http://localhost:8080/` in your browser.

## ☁️ Deploying to Google Cloud Run

The service can be deployed to Google Cloud Run in a single command utilizing the pre-configured source build script:

```bash
chmod +x deploy.sh
./deploy.sh
```

This script automatically:
1. Retrieves the active GCP project using the `gcloud` CLI.
2. Builds the container image using Cloud Build.
3. Deploys the container to Cloud Run in the `us-central1` region with public unauthenticated access (so Gemini Enterprise can initiate the handshake).
4. Prints the exact endpoints to use during registration.

## 🔌 Connecting to Gemini Enterprise

To register the deployed server as a **Custom MCP Datastore** in the **Vertex AI Search and Conversation** console:

1. Navigate to **Data stores** ➡️ **Create data store** ➡️ **Custom MCP Server**.
2. Fill in the connection form using your deployed Cloud Run URLs:

| Connection Profile Field | Value / Format |
|---|---|
| **MCP Server URL** | `https://<YOUR_CLOUD_RUN_URL>/mcp` |
| **Authorization URL** | `https://<YOUR_CLOUD_RUN_URL>/oauth/authorize` |
| **Token URL** | `https://<YOUR_CLOUD_RUN_URL>/oauth/token` |
| **Client ID** | `mock-client-id` |
| **Client Secret** | `mock-client-secret` |
| **Scopes** | `read write` (or any placeholder string) |

3. Save the Data Store and link it to your target Chat Application.

## 🔒 Moving to Production: Real OAuth 2.0 Integration

The OAuth provider implemented in `oauth.py` is a **mock simulation** designed exclusively to satisfy Gemini Enterprise's structural requirements during registration without requiring an external Identity Provider (IdP).

For a secure production deployment, this mock OAuth flow must be replaced with a real, enterprise-grade Identity Provider (IdP) such as **Okta**, **Microsoft Entra ID (Azure AD)**, or **Google Cloud Identity**.

### Production Authentication Steps:

1. **Register Gemini Enterprise in your IdP**:
   Register Gemini Enterprise as a secure client application in your corporate Identity Provider.
   * Set the **Redirect URI** to Google's official redirect endpoint: `https://vertexaisearch.cloud.google.com/oauth-redirect`
   * Obtain the production `Client ID` and `Client Secret`.

2. **Configure the Data Store with your IdP Endpoints**:
   When registering the Custom MCP Data Store in the Google Cloud Console, do not use the Cloud Run OAuth URLs. Instead, populate the authentication fields with your IdP's real endpoints:
   * **Authorization URL**: Your IdP's authorization endpoint (e.g., `https://company.okta.com/oauth2/default/v1/authorize`)
   * **Token URL**: Your IdP's token exchange endpoint (e.g., `https://company.okta.com/oauth2/default/v1/token`)
   * **Client ID & Client Secret**: The credentials generated by your IdP in Step 1.

3. **Secure your MCP Server (Egress Validation)**:
   Remove the mock `oauth.py` router from `main.py`. Secure the `/mcp` endpoint by adding a token validation middleware (such as Google IAP or standard JWT verification) to verify that incoming requests contain a valid `Authorization: Bearer <JWT_TOKEN>` signed and issued by your corporate IdP.

For detailed instructions, please refer to the official Google Cloud documentation:
* [Set up a custom MCP server data store](https://cloud.google.com/gemini/enterprise/docs/connectors/custom-mcp-server/set-up-custom-mcp-server)
* [Authenticate custom tools and resources in Agent Registry](https://cloud.google.com/agent-registry/docs/authenticate-toolsets)

## 🚨 Crucial: Activating the Actions in the Console

By default, custom MCP tools are registered in a **`Disabled`** state. The chat app will not have access to them until you explicitly enable them:

1. In the Google Cloud Console, navigate to your created **ServiceNow BYOMCP** Data Store details page.
2. Click the **`Actions`** tab in the left menu.
3. Click the checkbox at the top-left of the table header to **select all 6 ServiceNow tools**.
4. Click the **`Enable`** button that appears at the top.
5. The Status column must show a green **`Enabled`** status for all tools.
6. Now, open a **New Chat Thread** in Gemini Enterprise, authenticate/link your user session, and run the query:
   > *"Use ServiceNow BYOMCP to list my recent incidents."*

## ⚙️ Implicit Approval for Read-Only Actions

By default, Gemini Enterprise prompts the user for explicit consent before running *any* Custom Action tool. To optimize the user experience, this repository is pre-configured to support **Implicit Approval (silent tool calling) for Read-Only actions** while keeping write/delete actions safely secured behind explicit prompts.

### How It Works under the Hood:
The `tools/list` JSON-RPC responder in `main.py` automatically injects standard **Model Context Protocol (MCP) Tool Annotations** into the tool discovery payload:

*   **Read-Only Tools** (`list_incidents`, `query_incident`, `get_ticket_comments`):
    Annotated with `"readOnlyHint": true` and `"destructiveHint": false`. 
    *Gemini Enterprise executes these tools completely silently without any popups.*
*   **Write / Modify Tools** (`add_ticket_comment`, `delete_ticket_comment`, `update_ticket_fields`):
    Annotated with `"readOnlyHint": false` and `"destructiveHint": true`.
    *Gemini Enterprise enforces explicit security boundaries, prompting the user for confirmation before running these actions.*
